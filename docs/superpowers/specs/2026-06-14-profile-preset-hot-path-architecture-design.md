# Архитектура быстрого изменения preset-а и стратегии

## Задача

Смена preset-а и готовой стратегии сейчас работает, но один маленький клик запускает слишком широкую цепочку: GUI обновляет выбранную строку, сервис меняет preset, runtime применяет preset к `winws`, а страницы заново пересобирают список profile-ов и сводку.

Нужно уменьшить лишнюю работу, не ухудшая удобство GUI:

- выбранная стратегия должна подсвечиваться сразу;
- быстрые клики не должны зависать;
- runtime должен применить последнюю выбранную версию;
- список profile-ов не должен полностью пересобираться без причины.

## Текущая цепочка

При смене стратегии:

```text
profile_setup_page.py
  сразу меняет выбранную строку

ProfileStrategyApplyWorker
  вызывает применение стратегии в фоне

ProfilePresetService
  меняет модель preset-а и сохраняет файл

PresetFileService
  публикует событие изменения preset-а

PresetRuntimeCoordinator
  применяет изменённый preset к winws

profile/preset UI
  пересчитывает список profile-ов и сводку
```

Главная архитектурная проблема: событие "preset изменился" слишком общее. Оно нужно для runtime и для GUI, но разные потребители получают его как повод пересчитать почти всё.

## Целевое правило

Изменение должно возвращать план, а не просто будить все слои.

План отвечает на простые вопросы:

- какой preset изменился;
- какой profile изменился;
- изменилась ли структура списка profile-ов;
- нужно ли обновить только одну строку;
- нужно ли пересчитать сводку;
- нужно ли применить preset к запущенному `winws`.

GUI остаётся тонким: он показывает состояние и вызывает готовые действия. Сервисы решают, что реально изменилось.

## Этап 1. Явный результат изменения стратегии

Добавить более точный результат для применения стратегии.

Сейчас `StrategyApplyResult` уже говорит, применилось ли действие. Нужно расширить смысл результата без ломки старого поведения:

- `profile_key` остаётся ключом изменённого profile;
- добавить признак, что структура списка не изменилась;
- добавить признак, что изменилась только стратегия одного profile;
- сохранить текущий быстрый локальный отклик GUI.

После этого страница настройки profile сможет обновлять свой payload и сводку, не требуя полного пересчёта списка при каждом клике.

## Этап 2. Развести обновление GUI и применение runtime

`PresetRuntimeCoordinator` остаётся владельцем применения preset-а к `winws`.

Но событие сохранения preset-а не должно автоматически означать "всем страницам срочно пересобрать всё". Для GUI нужен отдельный путь:

```text
изменился один profile
  обновить строку profile, setup текущего profile и сводку

изменилась структура preset-а
  пересобрать весь список profile-ов

сменился preset
  пересобрать весь список profile-ов
```

Runtime при этом продолжает получать сигнал, что активный preset надо применить заново. Если пользователь быстро меняет стратегии, применяется последняя версия после короткой паузы.

## Этап 3. Узкий кэш вместо полного сброса

Кэш нужно разделить по смыслу:

- разобранный выбранный preset;
- список profile-ов;
- setup конкретного profile;
- краткая сводка для главной страницы.

При смене стратегии в одном profile сбрасывается только setup этого profile и сводка. Полный список сбрасывается только при добавлении, удалении, перестановке profile-ов или смене preset-а.

Это должно уменьшить частые вызовы `profile_list_item.build`, который сейчас даёт заметный пик ЦП.

## Что не меняем

- Не меняем внешний вид страниц.
- Не убираем мгновенную подсветку выбранной стратегии.
- Не меняем формат preset-файлов.
- Не меняем правила запуска `winws`.
- Не добавляем старые fallback-пути ради совместимости.

## Проверки

Нужны точечные тесты:

- смена стратегии возвращает результат "изменился один profile";
- быстрая серия смен стратегий применяет последнюю;
- при смене стратегии не требуется полный rebuild списка, если структура profile-ов не изменилась;
- при смене preset-а полный rebuild остаётся;
- runtime всё ещё применяет изменённый активный preset;
- текущие проверки архитектуры проходят.

## Порядок реализации

1. Сначала добавить тесты на новый контракт результата и отсутствие лишнего полного пересчёта.
2. Затем расширить результат `StrategyApplyResult`.
3. После этого научить GUI использовать точечное обновление, если структура списка не изменилась.
4. Отдельным маленьким шагом сузить инвалидирование кэша.
5. В конце проверить живую цепочку по логам `C:\Zapret\Dev\logs`.
