# Архитектура быстрого изменения preset-а и стратегии

## Задача

Смена preset-а и готовой стратегии сейчас работает, но один маленький клик запускает слишком широкую цепочку: GUI обновляет выбранную строку, сервис меняет preset, runtime применяет preset к `winws`, а страницы заново пересобирают список profile-ов и сводку.

Нужно уменьшить лишнюю работу, не ухудшая удобство GUI:

- выбранная стратегия должна подсвечиваться сразу;
- быстрые клики не должны зависать;
- runtime должен применить последнюю выбранную версию;
- список profile-ов не должен полностью пересобираться без причины.

## Текущая цепочка

При смене стратегии:

```text
profile_setup_page.py
  сразу меняет выбранную строку

ProfileStrategyApplyWorker
  вызывает применение стратегии в фоне

ProfilePresetService
  меняет модель preset-а и сохраняет файл

PresetFileService
  публикует событие изменения preset-а

PresetRuntimeCoordinator
  применяет изменённый preset к winws

profile/preset UI
  пересчитывает список profile-ов и сводку
```

Главная архитектурная проблема: событие "preset изменился" слишком общее. Оно нужно для runtime и для GUI, но разные потребители получают его как повод пересчитать почти всё.

## Целевое правило

Изменение должно возвращать план, а не просто будить все слои.

План отвечает на простые вопросы:

- какой preset изменился;
- какой profile изменился;
- изменилась ли структура списка profile-ов;
- нужно ли обновить только одну строку;
- нужно ли пересчитать сводку;
- нужно ли применить preset к запущенному `winws`.

GUI остаётся тонким: он показывает состояние и вызывает готовые действия. Сервисы решают, что реально изменилось.

## Границы ответственности

Главное правило: маленькое действие не должно рассылать общий сигнал всем слоям, если результат действия уже понятен.

Роли слоёв:

- GUI-страницы принимают клик, сразу показывают локальный отклик и вызывают готовое действие. Они не решают, как устроен preset и надо ли пересобирать весь список.
- `ProfilePresetService` владеет смыслом profile-ов и стратегий. Он решает, что именно изменилось: стратегия одного profile, содержимое profile, структура preset-а или выбранный preset.
- `PresetFileService` владеет записью preset-файла и файловым событием "preset изменился". Он не должен быть единственным языком для всех изменений GUI.
- `PresetRuntimeCoordinator` владеет применением активного preset-а к `winws`. Он не собирает payload для GUI и не решает, какие строки страницы обновлять.
- Кэш хранит только готовые результаты чтения и сборки. Он ускоряет повторный показ, но не является источником истины и не должен скрывать широкую пересборку.

Такой порядок сохраняет удобство GUI: клик сразу виден пользователю, но тяжёлая работа остаётся в worker/service-слоях.

## Типы изменений

Нужно различать минимум четыре типа:

```text
strategy_only
  поменялась только готовая стратегия одного profile
  можно обновить setup текущего profile, строку profile и сводку
  полный список profile-ов не нужен

profile_content
  поменялись match-строки, фильтр, raw profile или enabled-состояние
  обычно нужна точечная замена строки и сводка

preset_structure
  profile добавлен, удалён, продублирован или переставлен
  нужен полный список profile-ов

preset_selection
  выбран другой preset
  нужен payload выбранного preset-а
```

Если код не знает тип изменения, он может временно идти по безопасному полному reload. Но новый код должен стараться вернуть точный тип, а не прятаться за общим событием.

## Правило маленького радиуса

Каждое действие должно заранее говорить, насколько далеко расходится его эффект:

- смена стратегии существующего profile обновляет только этот profile, сводку и runtime;
- смена preset-а меняет весь выбранный набор, поэтому полный reload списка разрешён;
- добавление, удаление, дублирование и перестановка profile-ов меняют структуру, поэтому полный reload списка разрешён;
- ручное редактирование raw preset-а, импорт, сброс и внешнее изменение файла остаются широкими событиями, потому что код не всегда знает точную разницу;
- кэш можно использовать для быстрого открытия уже прогретого preset-а, но нельзя считать кэш доказательством, что rebuild не нужен.

## Запреты

- Не переносить разбор preset-а в GUI.
- Не заставлять GUI самому сравнивать старый и новый `.txt` preset.
- Не добавлять второй источник истины рядом с `ProfilePresetService`.
- Не делать кэш обязательным условием правильного поведения.
- Не отправлять общий `preset_content_revision` как единственную причину для полного rebuild, если сервис уже вернул точный результат действия.
- Не отключать runtime-apply ради быстрого интерфейса: если активный preset изменился, `winws` должен получить последнюю версию.
- Не оставлять два параллельных пути для одного действия после завершения этапа.

## Этап 1. Явный результат изменения стратегии

Добавить более точный результат для применения стратегии.

Сейчас `StrategyApplyResult` уже говорит, применилось ли действие. Нужно расширить смысл результата без ломки старого поведения:

- `profile_key` остаётся ключом изменённого profile;
- добавить поле `change_kind`;
- для смены готовой стратегии возвращать `change_kind="strategy_only"`;
- добавить признак, что структура списка profile-ов не изменилась;
- добавить признак, что payload одного profile изменился;
- добавить признак, что runtime должен применить активный preset;
- сохранить текущий быстрый локальный отклик GUI.

После этого страница настройки profile сможет обновлять свой payload и сводку, не требуя полного пересчёта списка при каждом клике.

## Этап 2. Развести обновление GUI и применение runtime

`PresetRuntimeCoordinator` остаётся владельцем применения preset-а к `winws`.

Но событие сохранения preset-а не должно автоматически означать "всем страницам срочно пересобрать всё". Для GUI нужен отдельный путь:

```text
изменился один profile
  обновить строку profile, setup текущего profile и сводку

изменилась структура preset-а
  пересобрать весь список profile-ов

сменился preset
  пересобрать весь список profile-ов
```

Runtime при этом продолжает получать сигнал, что активный preset надо применить заново. Если пользователь быстро меняет стратегии, применяется последняя версия после короткой паузы.

## Этап 3. Узкий кэш вместо полного сброса

Кэш нужно разделить по смыслу:

- разобранный выбранный preset;
- список profile-ов;
- setup конкретного profile;
- краткая сводка для главной страницы.

При смене стратегии в одном profile сбрасывается только setup этого profile и сводка. Полный список сбрасывается только при добавлении, удалении, перестановке profile-ов или смене preset-а.

Это должно уменьшить частые вызовы `profile_list_item.build`, который сейчас даёт заметный пик ЦП.

## Этап 4. Удалить широкие fallback-ветки

После того как точечный путь покрыт тестами, надо убрать старые широкие ветки там, где они больше не нужны.

Цель не в том, чтобы любой ценой запретить полный reload. Полный reload остаётся для смены preset-а, изменения структуры и ошибок stale-состояния. Но путь `strategy_only -> полный rebuild списка` должен перестать быть обычным рабочим сценарием.

## Что не меняем

- Не меняем внешний вид страниц.
- Не убираем мгновенную подсветку выбранной стратегии.
- Не меняем формат preset-файлов.
- Не меняем правила запуска `winws`.
- Не добавляем старые fallback-пути ради совместимости.

## Проверки

Нужны точечные тесты:

- смена стратегии возвращает результат "изменился один profile";
- `strategy_only` не вызывает полный rebuild списка profile-ов;
- быстрая серия смен стратегий применяет последнюю;
- при смене стратегии не требуется полный rebuild списка, если структура profile-ов не изменилась;
- при смене preset-а полный rebuild остаётся;
- runtime всё ещё применяет изменённый активный preset;
- текущие проверки архитектуры проходят.

## Порядок реализации

1. Сначала добавить тесты на новый контракт результата и отсутствие лишнего полного пересчёта.
2. Затем расширить результат `StrategyApplyResult`.
3. После этого научить GUI использовать точечное обновление, если структура списка не изменилась.
4. Отдельным маленьким шагом сузить инвалидирование кэша.
5. Удалить широкие fallback-ветки, которые стали не нужны.
6. В конце проверить живую цепочку по логам `C:\Zapret\Dev\logs`.
